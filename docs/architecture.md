# Obsidian Multiplayer: Architecture & Design

## Goals

- Real-time collaborative editing across Obsidian vaults
- **Zero-knowledge server**: the relay never sees plaintext document content or room keys
- **Zero-trust peer identity**: peers can verify each other's identity without trusting the relay
- Free for individuals, commercially licensable for teams
- Self-hostable for enterprises and power users

---

## Transport Layer

Yjs supports multiple providers simultaneously. The plugin activates both:

- **y-websocket** (encrypted relay) — guarantees availability; a new peer can sync even if no other peers are currently online
- **y-webrtc** — direct P2P when peers are online; lower latency, no relay bandwidth cost

Yjs merges updates from both providers transparently.

### Why not pure WebRTC?

Pure P2P cannot bootstrap a document when no peers are online. IndexedDB provides local persistence, but a peer who has never synced a room cannot load it without at least one online peer. The WebSocket relay solves this by storing (encrypted) Yjs update history.

### Why not pure WebSocket?

A standard y-websocket server sees plaintext. Encrypting updates client-side before they reach the relay preserves the zero-knowledge property while keeping availability.

---

## Encryption

All Yjs updates are encrypted client-side with AES-256-GCM before leaving the device. The relay stores and relays ciphertext only.

```
Room key (AES-256, user-held, never sent to relay)
    ↓
encrypt(Yjs update binary)
    ↓
WebSocket relay ← ciphertext only
    ↓
Peers decrypt locally
```

The existing `pwManager.ts` AES-256-GCM implementation is reused for this layer. The per-room `encPw` field in `SharedTypeSettings` becomes a wrapped AES room key rather than an encrypted password string — structurally identical.

## Identity and Key Infrastructure

### Per-User Keypair

On first OAuth login, the plugin generates an asymmetric keypair (e.g. X25519 or P-256) and registers the public key fingerprint with the auth server. The private key is stored locally, encrypted with the master-password-derived key (existing scrypt + AES-GCM in `pwManager.ts`).

This keypair is the root of trust for the user. Room keys are never transmitted as plaintext — they are always wrapped (asymmetrically encrypted) to a recipient's public key.

### Room Keys

Each room has a random AES-256 room key generated by the creator. It is stored on the relay as a map of `{ userId → encrypt(roomKey, userPubkey) }`. Each member fetches and decrypts their copy using their private key. The relay cannot decrypt any entry in the map.

## Authentication and Authorization

### Auth Server

A lightweight auth server (deployable as a Cloudflare Worker) handles:

- OAuth 2.0 PKCE flow (Google, GitHub, etc.)
- Public key registration and lookup by user identity
- Issuance of short-lived, room-scoped JWTs

The relay validates JWTs on WebSocket connect and enforces room membership. It does not need to understand room contents.

### Peer Identity Verification (Zero-Trust Layer)

For stronger peer-to-peer trust, peers can verify each other's identity without relying solely on the relay:

1. Each user's JWT includes their public key fingerprint (issued by the auth server)
2. A peer challenges the remote with a nonce
3. Remote signs the nonce with their private key
4. Local peer verifies: signature matches pubkey in JWT, JWT is auth-server-signed

This proves both "I own this private key" and "this key is bound to an OAuth-verified identity" — without the auth server being online at verification time.

## Invitation and Key Delivery

Key material never appears in any URL, message, or log.

### Known User (already registered)

1. Owner enters invitee's email or username
2. Auth server returns invitee's public key fingerprint
3. Owner's plugin wraps the room key with invitee's public key
4. Wrapped key is uploaded to relay, keyed to invitee's user ID
5. Invitee's plugin fetches and decrypts on next login — no simultaneous presence required

### New User (not yet registered)

1. Owner generates a short-lived, single-use invite token (room ID + auth token — no key material)
2. Token is sent to invitee via any channel (link, email, etc.)
3. Invitee clicks link → OAuth login → keypair generated and registered
4. Relay delivers the pre-wrapped room key immediately (owner wrapped it speculatively, or wraps it now)
5. Invite token is burned on use — forwarding the link is harmless

## Room Administration and Key Rotation

### Admin Set

"Owner" is replaced by an **admin set**. Any admin can add/remove members, promote/demote admins, and initiate key rotation. Rooms above two members should have at least two admins (enforced by relay).

### Key Rotation

When a member is removed (or an admin initiates rotation for any reason):

1. Initiating admin generates a new AES room key locally
2. New key is wrapped with each remaining member's public key
3. New `wrappedKeys` map is uploaded to relay
4. Relay marks the old key epoch as rotated
5. Each remaining member fetches their wrapped copy on next connect

Members do not need to be online simultaneously. The removed member cannot decrypt updates from the new epoch, even with a copy of the relay's stored ciphertext.

### Admin Recovery

If all admins become unavailable, any member may request admin elevation after a configurable inactivity timeout (default: 30 days). The request is visible to all remaining members. This prevents permanent room lockout without enabling silent takeover.

## Infrastructure and Deployment

### Cloudflare Stack (Default Hosted Instance)

| Component | Cloudflare Primitive |
|---|---|
| WebSocket relay (per-room state) | Durable Objects |
| Auth server (JWT issuance, key registry) | Workers |
| Encrypted Yjs update storage | Durable Object storage |
| Global edge delivery | Automatic (Workers network) |

Economics: Durable Objects are priced per-request and per-GB-stored. Yjs updates are small binary diffs. A typical active user generates ~5–10k requests/day. Cloudflare's free tier (100k requests/day, 1GB storage) comfortably covers dozens of active individual users before any cost is incurred.

### Self-Hosting

The relay and auth server are published as open source. Users configure the plugin with their own relay URL. This enables:

- Enterprise compliance (data never leaves their infrastructure)
- Air-gapped deployments
- Community-operated instances

Self-hosting is the natural commercial upsell tier: organizations get the server code, deployment docs, and a commercial license that removes the hosted-instance limits.

### Pricing Tiers

| Tier | Description |

## Security Properties Summary

| Threat | Mitigation |
|---|---|
| Relay operator reads documents | AES-256-GCM client-side encryption; relay sees ciphertext only |
| Relay operator reads room keys | Room keys wrapped with asymmetric user keys; relay cannot decrypt |
| Leaked invite link | Links contain auth tokens only, no key material; tokens are single-use |
| Impersonation by peer | Zero-trust nonce-signing: peer proves private key possession + JWT-bound identity |
| Removed member reads future updates | Key rotation on removal; new epoch unreadable with old key |
| Admin disappears | Admin set (not single owner); member elevation after inactivity timeout |
| Replay attacks | Short-lived JWTs; Yjs update ordering via CRDT semantics |

## What Stays the Same

The existing client-side implementation is largely preserved:

- `pwManager.ts` — reused for local encryption of room keys and private key storage
- `SharedTypeSettings.encPw` — repurposed as wrapped AES room key (same structure)
- `y-indexeddb` — unchanged; provides local persistence and offline support
- `y-webrtc` — unchanged; still used for direct P2P sync
- Master password + scrypt key derivation — unchanged; protects local key storage
- User awareness (cursors, names, colors) — unchanged

Primary additions: per-user keypair, auth server, encrypted WebSocket relay, async key delivery.
